什么是 CSS 模块

根据这篇文章，CSS 模块是这样说的：

CSS 文件中的所有类名和动画名都在默认的局部范围内。

所以 CSS 模块不是一个官方规范也不能在在浏览器中实现，而是生成步骤中的一个进程(利用Webpack或Browserify)，更改类名和选择器（即有点像命名空间）。

它是什么样子的呢？下面你就会知道，首先，想一想 HTML 和 CSS 是如何正常工作的，在 HTML 中定义一个类是这样的：

<h1 class="title">An example heading</h1>

CSS 代码是这样的：

.title {
  background-color: red;
}

一旦 CSS 应用到该 HTML 文档，<h1>背景色将变成红色。我们并不需要处理 CSS 或 HTML。浏览器会知道这两种格式。
CSS 模块采用了不同的方法。而不是编写纯 HTML 代码，我们需要把所有的标记写进一个 JavaScript 文件，比如 index.js。下面是个例子（后面会有一些更实际些的例子）：

import styles from "./styles.css";

element.innerHTML = 
  `<h1 class="${styles.title}">
     An example heading
   </h1>`;

在我们生成期间，编译器会搜索我们已经导入的styles.css文件，然后通过 JavaScript 中写到的styles.title代码，再把.title编译出来。然后生成器会用一个新的字符串把这些处理进一个新的、独立的 HTML 和 CSS 文件中，以取代 HTML 类和 CSS 选择器类。

我们生成的 HTML 代码可能是这样的：

<h1 class="_styles__title_309571057">
  An example heading
</h1>

我们生成的 CSS 代码可能是这样的：

._styles__title_309571057 {
background-color: red;
}

类的属性和.title选择器完全消失了，被这个全新的字符串替换， 我们原来的 CSS 不会提交到浏览器。

正如Hugo Giraudel在他关于这个主题的教程中所说的：

[类]是动态生成的，独特的，并且映射到正确的样式上。

意思是样式被特定的模板风格所覆盖。如果我们有一个buttons.css文件，我们只导入到一个buttons.js模板中，.btn类内将无法被其他的模板（如forms.js）访问，除非我们专门导入它。

为什么我们想让 CSS 和 HTML 要这样做？

怎么使用 CSS 模块

在 CSS 模块中，要确保所有的样式都是单个的组件：

  1. 用在一个地方
  2. 只对那个组件生效

另外，每个组件都有真正的依赖性，比如：

import buttons from "./buttons.css";
import padding from "./padding.css";

element.innerHTML = `<div class="${buttons.red} ${padding.large}">`;

这种方法旨在修正CSS中的全局范围的问题。
你是否曾经由于没时间就草草的写 CSS代码，还不考虑后果吗？

你是否在样式表的底部敲过一些随机字符和垃圾代码，还打算以后重新组织起来，但没做？

你是否遇到过你不确定你的样式是不是完全正确，甚至它们真的被用到了吗？

你有没有想过，你可以不用打散代码就能能摆脱一些不需要的样式？想知道如何让样式能够有独立性？还是应该直接改写样式？

这些问题都是让人感到头疼，还有复杂的项目，蛋疼的工期神马的。

用到 CSS 模块和本地默认范围的概念，这个问题是可以避免的。你总是在写样式的时候考虑后果。

举例来说，如果在 HTML 中使用random-gross-class,不会把它当成一个 CSS 模块式的类，样式将不会被生效，因为 CSS 选择器将把它转换为._style_random-gross-class_0038089。

composes关键字

比如说我们有这样一个样式表，叫type.css：

.serif-font {
  font-family: Georgia, serif;
}

.display {
  composes: serif-font;
  font-size: 30px;
  line-height: 35px;
}

我们会把类定义成这样：

import type from "./type.css";

element.innerHTML = 
  `<h1 class="${type.display}">
    This is a heading
  </h1>`;

在标记中结果是这样：

<h1 class="_type__display_0980340 _type__serif_404840">
  Heading title
</h1>

这些类已通过使用composes关键字绑定到元素中，从而避免了一些类似的问题，比如 Sass 的@extend。
我们甚至可以在一个单独的 CSS 文件中的组成特定的类：

.element {
  composes: dark-red from "./colors.css";
  font-size: 30px;
  line-height: 1.2;
}

不需要 BEM

当我们用 CSS 模块的时候我们不需要使用 BEM。有两个原因：

  1. 易解析：对开发者来说，type.display这样的代码比 BEM-Y 的.font-size__serif--large更为清晰。这比用更长的 BEM 选择器可能更容易理解。
  2. 局部作用域：假设我们一个模块中有一个.big类，改变了font-size。在另一个相同的.big类，以不同的量增加了padding和font-size。没关系！它们不会有冲突，因为样式作用域是有意义的。即使同一个模块都导入样式表，编译器会使专门为该类定义一个名称。
    换句话说，CSS 模块中特殊问题消失了。

很酷吧？

这些只是写 CSS 模块的的一部分优点。
